# Полный аудит проекта Beat Buddy (Stockley)

**Дата:** 13 февраля 2026  
**Версия:** 2.1.0

---

## 1. Обзор проекта

**Beat Buddy** — веб-приложение для тренировки чувства ритма у детей 6–12 лет. Ребёнок под руководством педагога слушает метроном в наушниках и бьёт по датчику на перчатке, получая визуальную обратную связь через анимацию.

**Стек:** Vanilla JS (ES modules), PixiJS 7.4.2, Web Audio API, CSS3, HTML5. Без фреймворков, без бэкенда.

---

## 2. Архитектура

### 2.1 Общая схема

```
MetronomeEngine → (beatTimes) → RhythmAnalyzer
SensorManager → (hitTime) → RhythmAnalyzer → (zone) → Animator
app.js — координатор: связывает модули, управляет экранами

Animator (владеет PixiJS App, ticker, lifecycle)
  └── location = LocationRegistry.create(id)
        └── { init(container, w, h), onBeat(phase, beatsFromStart), onHit(zone), destroy() }
```

### 2.2 Потоки данных

| Источник | Данные | Приёмник |
|----------|--------|----------|
| MetronomeEngine | firstBeatTime (performance sync) | RhythmAnalyzer, Animator |
| SensorManager | hitTime (timestamp) | app.js → RhythmAnalyzer |
| RhythmAnalyzer | zone (perfect/good/miss), deviation | Animator, app.js (streak, speech) |
| LocationRegistry | locationId | Animator, app.js |

### 2.3 Оценка архитектуры

**Сильные стороны:**
- Чёткое разделение ответственности: метроном, сенсор, анализ, визуал
- LocationRegistry — расширяемая система локаций без жёсткой связанности
- app.js как единственный координатор — понятный entry point
- Синхронизация через `firstBeatTime` (AudioContext → performance) — корректное решение для точного анализа

**Слабые стороны:**
- app.js перегружен (~590 строк): UI, бизнес-логика, координация в одном файле
- Нет явного слоя domain: бизнес-правила (пороги, streak) размазаны по модулям
- Отсутствие unit-тестов — критичная логика (RhythmAnalyzer, MetronomeEngine) не покрыта
